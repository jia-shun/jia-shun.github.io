IDEA中Git使用教程以及Git详解

2017-11-26 • 分类: Java • 标签: java

一：IDEA下的Git使用

本教程默认电脑已安装好Git并和远程仓库配置好了SSH连接。如果想详细了解Git，可参考本文的下半部分内容。

开发环境使用Mac，Windows下基本相似。远程仓库使用GitHub，和公司自己的Git服务器使用流程基本一样。

一：在Idea中配置Git

1：IntelliJ IDEA —>Performance—>Version Control—>Git，将自己安装git的文件路径填入Path to Git executable。

2：IntelliJ IDEA —>Performance—>Version Control—>GitHub，登录你的GitHub账户，输入用户名和密码。

二：将项目share到GitHub远程仓库

IntelliJ IDEA —>VCS—>Import into Version Control—>Share Project On GitHub

输入项目仓库名字，点击share

此刻登录你的GitHub，发现已经有了一个新的远程仓库，名字就是你刚才share的New Repository name：

此时可以在页面中点击Branch新建一个dev的分支，以后开发环境在dev分支中即可。

此刻整个远程仓库项目建立完成。

三：协作开发流程

1：此刻所有的开发人员都可以去GitHub（或者公司Git服务器）克隆这个项目：

拷贝项目的远程仓库地址，通过Idea导入项目。

2：点击右下角，将分支切换至dev，进行代码开发并提交。

当你对代码进行了修改或者完成了新代码的编写，就需要上传至Git服务器。上传之前先进行Pull（拉取）命令，相当与原来SVN的update。

随后进行Commit。但要注意一点，此刻必须要写Commit Message，就是此次对项目做了哪些工作的注释文件，否则不能Commit。

如果确认无误，可直接执行Commint and Push。

关于Commit和Push的区别，可以仔细阅读本文后半部分Git的详细介绍。

3：分支合并

因为此时是在dev模式下进行的开发，所以最终要将dev的修改合并到master主分支中。

此时切换到master分支中，执行Merge Changes。

基本上掌握这些就可以完成Git在Idea中的基本使用。还要很多，分支冲突的解决，版本的回滚等，可以参考下面的Git详细介绍了解。

二：Git详解

一：Git 个人全局信息设置：

git config –global user.name “jiashun”

git config –global user.email “jiashun@outlook.com”

二：Git基础命令：

创建git本地仓库：git init

将自己的项目放入git仓库文件夹下。

将项目添加到仓库：git add 文件名

将项目提交到仓库：git commit -m “本次提交的说明”

实时查看git状态：git status

如果git status告诉你有文件被修改过，查看git具体修改的内容： git diff

三：Git版本穿梭：

实际工作中，版本控制系统肯定有某个命令可以告诉我们历史记录，在Git中，使用命令： git log

查看历史版本的时候会看到一大串类似3458adv…882e10这样的commit id（版本号）。

在Git中，使用HEAD表示当前版本，上一个版本就是HEAD^，上上个版本就是HEAD^^，版本多了可以使用HEAD~10（前10个版本）

Git允许我们在版本的历史之间穿梭，使用命令git reset –head commit_id

穿梭前，用git log可以查看提交历史，以确定回到哪个版本。

要重返未来，用git reflog查看历史命令，以便确定要回到未来的哪个版本。

Git和其他版本控制系统如SVN的一个不同之处就是有暂存区的概念。

四：几个基础名词：

工作区（Working Directory）：就是电脑上我们的项目目录。

版本库（Repository）:工作区有一个隐藏目录.git，这个不算工作区，而是git的版本库。

git的版本库存了很多东西，其中最重要的称为stage(暂存区)，还有Git为我们自动创建的第一个分支master，以及指向master的一个指针：HEAD


总结将文件往Git版本库添加的过程：

1：用 git add 命令将文件添加进去，实际上就是把文件修改添加到暂存区。

2：使用 git commit提交更改，实际上就是将暂存区的内容提交到当前分支。

你可以简单理解为：需要将提交的文件修改统统放到暂存区，然后，一次性提交暂存区的所有修改。

暂存区是个很重要的概念。

理解了暂存区的概念，就会发现为什么Git比其他版本控制系统设计的优秀，因为Git跟踪并管理的是修改，而非文件。

每次修改让后git add是将修改的版本放入暂存区，然后执行git commit才会提交到真正的版本库。

五：撤销修改：

当你改乱了工作区某个文件的内容，想直接丢弃工作区的修改时，使用命令 git checkout – 文件名

当你改乱了工作区某个文件并提交到了暂存区，修丢弃修改。第一步先：git reset HEAD 文件名，第二步：按上面的操作，使用git checkout –文件名 。

如果修改已经提交到版本库，只能使用版本回退。

六：删除文件：

命令git rm用于删除一个文件。

七：远程仓库：

远程仓库是一个很重要的概念。

Git是分布式版本控制系统，同一个Git仓库，可以分布到不同的机器上。最早，肯定只有一台机器有一个原始版本库，此后，别的机器可以“克隆”这个原始版本。

实际情况是这样：公司找一台电脑充当服务器，每天24小时开机，其他每个人都从这个“服务器”仓库克隆一份到自己电脑，并且各自把各自的提交推送到服务器仓库，也从服务器仓库中拉取别人的提交。

这个世界上有一个神奇的网站叫GitHub，就是提供Git仓库托管服务的。只要注册一个账号就可以免费获得Git远程仓库。而获得的这个git仓库和自己搭建一台运行Git的服务器功能是一样，它就是一个远程仓库。以下的操作完全可以将GitHub远程仓库理解为公司搭建的私有Git仓库。

由于你的本地Git仓库和GitHub仓库之间传输是通过SSH加密的，所以需要一些设置：

生成SSH Key：ssh-keygen -t rsa -c “你的email地址”

一路回车正常会在主目录找到.ssh目录，里面有id_rsa（私钥）和id_rsa.pub（公钥）两个文件。私钥要保存好不要透漏出去，公钥可以放心告诉别人。登录github，在设置中，填上生成的公钥，这样本地仓库和github就建立了安全连接。如果公司有Git服务器，将公钥给运维人员，将其添加到Git服务器即可。

”有了远程仓库，妈妈再也不用担心我的硬盘了“

八：添加远程仓库：

在本地创建了一个本地仓库后，又想在GitHub创建一个Git仓库，并且让这两个仓库进行远程同步，这样，GitHub上的仓库既可以作为备份，又可以让其他人通过该仓库来写作。这样就实现了一个开发团队协作的流程。

首先，登录GitHub，然后点击“Create a new repo”创建一个新的仓库。

要关联一个远程库，使用命令:git remote add origin git@server-name:path/repo-name.git

关联后，使用命令git push -u origin master第一次推送master分支的所有的内容。

此后，每次本地提交后，只要有必要，就可以使用 git push origin master推送最新修改。

当然正常来说，我们从零开始开发，最好的方式先创建远程库，然后，从远程库克隆。

九：分支管理：

分支管理也是Git中一个非常重要的概念和功能。

分支在实际中的作用：假设你准备开发一个新功能，但是需要两周才能完成，第一周你写了50%的工作，如果立刻提交，不完整的代码库会导致别人不能干活了。如果等待代码全部写完再提交，又存在丢失每天进度的巨大风险。

现在有了分支就不用怕了，你创建了一个属于你自己的分支，别人看不到，还继续在原来的分支上正常工作，而你在自己的分支上干活，想提交就提交，直到开发完毕，再一次性合并到原来的分支，这样，既安全，又不影响别人工作。

十：创建与合并分支：

在版本回退中，每次提交，Git都会把他们串成一条时间线，这条时间线就是一个分支。截止到目前，只有一条时间线，也就是只有一个分支，我们称之为主分支，即master。HEAD严格来说不是指向提交，而是指向master，master才指向提交，所有HEAD指向的是当前分支。

当我们创建新的分支，例如dev时，Git新建了一个指针叫dev，指向master相同的提交，再将HEAD指向dev，表示当前分支在dev上。

这样Git创建一个分支dev，其实就是增加了一个dev指针，改变了HEAD的指向，工作区的文件没有任何变化。不过从现在开始，对工作区的修改和提升就是针对dev分支了，比如提交一个更新，dev指针就往前移动一步，而master指针不变。假如我们在dev上的工作完成了，就可以把dev合并到master上。Git合并的原理就是把master指向dev的当前提交，就完成了当前合并。

创建dev分支命令：git branch dev

切换到dev命令：git checkout dev

查看分支命令：git branch

会列出所有分支，当前分支前边会有*号

将dev分支的工作成果合并到master分支：git merge dev

删除dev分支：git branch -d dev

十一：解决冲突：

当两个分支同时对同一个文件进行了修改，执行分支合并的时候就会产生冲突，这个时候Git会告诉我们存在冲突的地方，必须手动解决冲突后再提交。

十二：分支策略：

在实际开发中，我们应该按照几个原则进行分支管理：

首先，master分支应该是非常稳定的，也就是仅用来发布新版本，平时不在上面干活。在dev分支上干活就可以了。所有的开发人员都在dev分支上干活，时不时的往dev分支上合并就可以了。

因此，多人协作的工作模式通常是这样：

1：首先，可以试图用 git push origin branch-name 推送自己的修改

2：如果推送失败，说明远程分支比你的本地更新，需要先用 git pull试图合并。

3：如果合并有冲突，则解决冲突，并在本地提交

4：解决掉冲突或者没有冲突，再用 git push origin branch-name 推送就能成功。
