认识 Spring Data JPA

一. ORM

概述：

面向对象编程讲究万物皆对象(object)，关系型数据库则是采用实体之间的关系(relation)连接数据。那么关系也可以用对象表达，这样就能用面向对象编程，来操作关系型数据库了。

简单说，ORM就是通过实例对象的语法，完成关系型数据库的操作。是“对象-关系映射”(Object/Relational Mapping)的缩写。

二：MyBatis && Hibernate

针对Java，使用最多的两个主流ORM框架就是MyBatis 和 Hibernate。

先看一个简单的对比：

MyBatis Hibernate
量级 轻，上手简单，插件多 重，功能齐全，精通较难
SQL自由度 高，提供更灵活的SQL编写方式 低。仅对基本的SQL语法有较好的支持。也支持自己手动写SQL
开发效率 低，需要维护SQL 高，DAO层开发简单，支持JPA
数据库无关性 差，所有SQL都是依赖数据库，需要针对特定的数据库维护SQL 高。高度解耦，封装了JDBC的实现，只需要在配置中指定数据库
缓存 机制相对较差 机制相对较好，可避免脏读
映射 实体与SQL的映射，将SQL所需的参数、返回的结果字段映射到指定的实体，属于半ORM，实际是SQL mapping 实体与数据库的映射，自动生成和执行SQL，完整的ORM
性能 高 用封装的性能会稍低一些，用Native查询方法性能与MyBatis差不多
使用场景 擅长复杂的查询，适用于分析型系统，计算机集群需要跨数据库事务时，MyBatis更灵活 面向对象的思维方式更有利于业务系统建模分析，适合单数据库，数据量小

关于这两个框架如何选型的讨论一直没有停止过。

如果你两个框架都使用过会明显发现在开发方式上的不同点。

MyBatis着力点在于实体与SQL之间的映射关系，在配置文件中基于xml的表达能力写SQL并将SQL所需的参数、返回的结果字段映射到指定的实体。

Hibernate对数据库结构提供了较为完整的封装，开发时只需要定义好和数据库表映射的实体对象，即可通过Hibernate提供的方法完成持久层操作，开发者甚至不需要对SQL熟练掌握(虽然SQL已经是程序员的基本技能)。

通过使用方式上的不同，可以看出MyBatis基于配置还是要手写SQL语句，属于半ORM。再返回我们最初的需求：开发者只面向对象编程，不用关心数据库操作。很显然，Hibernate貌似做的比较完善。

那是不是我们就用Hibernate就好了呢？

很遗憾，在国内，MyBatis的使用率远超Hibernate。我们聊聊原因。

在使用Hibernate的时候，我们查询的是实体类，而不再是数据库表。例如HQL：SELECT id, name FROM User。这里面的User是Java类，而不是数据库表user。ORM认为程序员使用OO的思维方式和关系数据库的思维方式不一样，所以在程序员使用纯对象的思维方式，查询的是实体对象，不再需要表、字段等关系的概念。一切看起来很美好。

但事实是，大多数情况下，使用HQL并没有比SQL简单。首先是HQL还是基于SQL，只不过偷换概念而已，反而使语言更加复杂和难以理解，其次是灵活性相对于SQL反而变得很差，表达能力也比SQL弱。尤其牵扯到多表关联查询之类的。

而MyBatis直接使用SQL语句查询，只不过将语句从最初的字符串拼接变成了写配置文件，将实体对象填入即可。简单理解为MyBatis是对JDBC的简单封装，从SQL的封装角度上看，MyBatis比Hibernate成功，SQL本不应该被封装和隐藏，而本身SQL又是程序员的基本技能，直接写就行了，还要去学着使用什么HQL，熟悉各种封装的ORM概念，多麻烦。

那是不是我们就用MyBatis就好了呢？

别着急，我们往下聊。

三： Spring-Data-JPA

1. JPA

JDK5.0新增了JPA (Java Persistence API) 协议：通过相关持久层注解来实现对象和关系型数据库表映射关系，并将Java项目运行期的实体对象，通过一种Session持久化到数据库中。

有了协议就有了标准规范，有了规范按照API去实现就可以了。

通过上图可以看出来Hibernate就是基于JPA的标准实现。

你是想做一个挑战者用JPA还是只做“搬砖者”用MyBatis？

2. Spring Data

Spring Data 作为Spring旗下的一个父项目，旨在统一和简化对各种类型持久化存储，而不拘泥于关系型数据库还是NoSQL存储。

无论是哪种存储，数据访问对象通常都会提供单一域对象的CRUD、排序、分页等操作。Spring Data则提供了基于这些层面的统一接口(CrudRepository，PagingAndSortingRepository)以及对持久存储的实现。

通过上图发现Spring Data JPA就是其中的一个子项目，是Spring针对JPA协议的实现。

3. Spring Data JPA

Spring Data JPA可以理解为JPA规范的再次封装抽象。而这方面Spring可以说是偷了一个懒，底层直接使用了Hibernate的JPA技术(其实就是Hibernate团队实现的)，引入JPQL查询语言。

由于Spring在Java中的地位，使用Spring Data JPA处理ORM相关的工作更加得心应手，再加上Spring本身又对Hibernate做了更完善的封装，使得整个项目更像一个整体，使用起来也比较方便，加快了开发效率。

总结一下：

  1. MyBatis会在项目中建立大量的xml配置文件，如果你是有代码洁癖的人会有些不能忍。
  2. Spring Data JPA基于Hibernate实现了JPA协议，基于Spring家族的体系使整个项目更整洁。
  3. Spring Data JPA封装了基础的ORM相关工作，对于不是特别复杂的数据库操作都有对应的接口实现，完全屏蔽了SQL。
  4. MyBatis的优势在于灵活性，尤其对于分库分表的大型项目，可以自由的写SQL比较方便，易于扩展。
  5. Spring Data JPA本身也支持基于@Native注解实现手写SQL，一些复杂的查询也提供了很多的封装都可以实现，只不过学习成本相对较高。

在可预见的时间内，我们的项目Spring Data JPA基本可以满足，基于此，我们选择Spring Data JPA。下面我们用一个简单的Quick Start感受一下使用Spring Data JPA开发的便捷性。

4. 乐观锁

乐观锁基于CAS的思想，在实际开发过程中经常使用。它没有加锁、没有阻塞，在多线程环境下及高并发的情况下CPU的利用率是最高的，吞吐量也是最大的。

而JPA协议也对乐观锁的操作做了规定：通过指定@Version字段对数据增加版本号，进而在更新的时候判断版本号是否有变化。如果没有变化则直接更新，否则就会抛出OptimisticLockException。我们用SQL表示一下乐观锁：

假设本次查询的version是1，在更新操作时会依据WHERE条件中的version值进行对比，如果version值没有变化则执行更新操作，如果更新失败代表version不再是1，中间过程应该是别的线程修改了version的值。多个线程之间并且不会相互覆盖，保证了数据的原子性。
